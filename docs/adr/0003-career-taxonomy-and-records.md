# ADR-0003: 직무 역량과 활동 유형을 분리하고 GitHub 기록을 블로그에 연결한다

- 상태: 승인
- 날짜: 2026-07-15

## 문제

글을 기술 주제만으로 분류하면 공모전·동아리·개인 프로젝트라는 맥락을 찾기 어렵다. 반대로 활동 유형을 기술 카테고리와 한 목록에 섞으면 PostGIS 성능 글이 `데이터·성능`인지 `개인 프로젝트`인지 하나만 선택해야 한다.

모든 작업 로그와 트러블슈팅을 블로그에 복사하면 글의 품질이 낮아지고 코드 변경과 원문 근거가 분리된다. 저장소에만 두면 채용 담당자가 여러 디렉터리를 직접 찾아야 한다.

## 채용 공고에서 확인한 기준

2026-07-15 기준 국내 백엔드·AI 플랫폼 공고를 확인했다. 반복해서 요구한 것은 특정 기술명 자체보다 다음 역량이었다.

- 문제 정의, 가설, 작은 검증과 측정 가능한 개선
- API·DB·비동기·분산 시스템의 기본기와 설계
- 배포, 모니터링, 장애 대응과 성능 최적화
- AI·RAG·Agent를 평가·운영 가능한 서비스로 연결하는 능력
- 기술 선택, 실패와 개인 기여를 명확하게 설명하는 문서화와 협업

근거로 확인한 대표 원문은 [당근 Backend 인턴](https://careers.daangn.com/jobs/role/7651256003/), [당근 커뮤니티 Backend](https://careers.daangn.com/jobs/role/7762062003/), [토스 AI Platform](https://toss.im/career/job-detail?job_id=7646941003), [토스 Messaging Platform](https://toss.im/career/job-detail?job_id=7719271003), [카카오 Search/ML/LLM](https://careers.kakao.com/jobs/P-14286) 공고다. 공고는 변경되거나 종료될 수 있으므로 사이트 문구에는 회사명을 나열하지 않고 실제 프로젝트 근거를 연결한다.

## 결정

글은 두 축으로 분류한다.

- 직무 역량: 백엔드·아키텍처, 인프라·운영, 데이터·성능, AI 시스템, 설계·개발 과정, 트러블슈팅·신뢰성
- 활동 유형: 개인 프로젝트, 공모전·대회, 동아리·연합활동, 팀·학교 프로젝트, 학습·기타

글 하나는 주 직무 역량 하나와 선택적 활동 유형 하나를 가진다. 세부 기술은 태그로 표현하고 프로젝트 관계는 별도로 유지한다.

기록은 GitHub 원본 + 블로그 선별 해설 + 프로젝트 상세 색인의 세 층으로 운영한다. 세부 규칙은 `docs/ENGINEERING_RECORDS.md`에 둔다.

## 대안

### 모든 분류를 단일 카테고리로 운영

기각. 기술 역량과 활동 맥락이 경쟁하고 글 탐색 목적이 불명확해진다.

### 모든 작업 로그를 블로그에 자동 복제

기각. 작은 변경이 공개 글을 압도하고, 원문 commit·PR과 떨어져 사실 검증이 어려워진다.

### GitHub에만 기록

기각. 코드에는 가장 가까우나 채용 담당자가 여러 저장소의 문서를 직접 탐색해야 한다. 블로그는 중요한 사례를 읽을 수 있는 형태로 편집하고 원문으로 되돌아갈 수 있게 한다.

## 결과와 재검토

카테고리 관리 화면은 분류 종류를 함께 편집하고, 글 목록은 두 그룹을 분리해 보여준다. 모든 프로젝트에는 활동 유형, 현재 상태 설명, 기록 운영 방식과 공개 원문 링크를 둔다.

글이 30편을 넘어 두 축만으로 탐색이 어려워지면 검색과 복수 직무 역량 분류를 검토한다. 여러 작성자가 생기면 GitHub main 직접 저장 대신 review branch와 PR 발행 흐름을 검토한다.
