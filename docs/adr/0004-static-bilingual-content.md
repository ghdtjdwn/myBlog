# ADR-0004: 영문 콘텐츠를 별도 정적 원문으로 운영한다

- 상태: 채택
- 결정일: 2026-07-15

## 맥락

해외 독자와 채용 담당자가 프로젝트를 읽을 수 있어야 하지만, 자동 번역 위젯은 문맥·기여 범위·기술 용어를 잘못 전달할 수 있다. 런타임 번역 API는 비용, 개인정보, 장애 의존성과 검색 색인 불확실성도 추가한다.

## 결정

한국어를 기본 URL로 유지하고 영어는 `/en/` 아래의 별도 정적 페이지로 제공한다. 프로젝트와 글은 `projects`/`projectsEn`, `posts`/`postsEn` 컬렉션으로 분리하며 같은 slug를 번역 쌍의 식별자로 사용한다. Keystatic에서 두 언어를 각각 편집하고, 공개 빌드는 draft를 언어별로 격리한다.

각 페이지는 자기 자신을 canonical로 선언하고 한국어·영어·x-default `hreflang`을 제공한다. UI 문자열, 날짜, 카테고리와 상태 이름도 언어별로 렌더링한다.

## 대안

- 브라우저 자동 번역 위젯: 구현은 빠르지만 기술 문맥과 SEO를 통제하기 어렵다.
- 요청 시 번역 API: 작성 비용은 줄지만 런타임 장애·비용·캐시·개인정보 경계가 생긴다.
- 하나의 파일에 두 언어 저장: 번역 누락은 보기 쉽지만 작성 화면과 스키마가 복잡해지고 본문 이식성이 낮아진다.

## 결과와 재검토

번역은 명시적인 검수 작업이지만 공개 결과가 재현 가능하고 Git 이력이 남는다. 프로젝트 번역 쌍은 CI에서 완전성을 검사한다. 영문 글은 검수를 마친 대표 글부터 선택적으로 공개하며, 글 수가 크게 늘어 번역 유지비가 병목이 되면 빌드 시 초안 생성과 사람 검수 흐름을 별도로 검토한다.
