# ADR-0002: Keystatic Git 기반 콘텐츠 관리자

- 상태: 승인
- 날짜: 2026-07-15

## 문제

Markdown 파일과 CLI만으로도 글을 쓸 수 있지만 공모전과 팀 활동 중에는 카테고리 탐색, 글 생성·수정·삭제, 이미지 추가, 공개·비공개 전환을 한 화면에서 처리해야 한다. 동시에 채용 포트폴리오인 만큼 글의 변경 이력과 코드·프로젝트 연결은 Git에 남아야 한다.

## 결정

Keystatic을 Astro에 통합하고 Production은 GitHub mode로 운영한다. 공개 페이지는 계속 prerender하고 `/keystatic`과 `/api/keystatic`만 Vercel Function으로 실행한다. 카테고리는 별도 YAML 컬렉션이며 글이 이를 참조한다. 새 글과 프로젝트는 기본적으로 초안이다.

## 이유

- 기존 Markdown/MDX와 Content Collections를 그대로 사용한다.
- 관리 화면에서 글·프로젝트·카테고리·ADR·트러블슈팅 CRUD가 가능하다.
- 편집 결과가 GitHub 커밋이므로 변경 이력, 복구, CI와 자동 배포가 한 흐름에 남는다.
- 독자 요청에는 서버나 DB가 필요 없고 관리자 인증 경로만 동적으로 운영한다.

## 대안

### 직접 만든 관리자와 데이터베이스

기각. 인증, 세션, 권한, 에디터, 이미지 저장, 백업과 마이그레이션을 모두 직접 운영해야 한다. 블로그의 차별점은 편집기 재구현보다 근거 있는 기술 기록에 있다.

### Sanity

보류. 강한 편집 경험과 구조화 콘텐츠를 제공하지만 콘텐츠 원본이 별도 Content Lake에 생기고 현재 Git 기반 검증·이식성에 추가 동기화가 필요하다. 여러 편집자나 실시간 협업이 실제 요구가 되면 재검토한다.

### Decap CMS

보류. Git 기반이라는 장점은 같지만 GitHub 인증 중계와 편집 경험을 별도로 구성해야 한다. 현재 Astro 통합과 필드 스키마에는 Keystatic이 더 직접적이다.

### Velog·Hashnode

주 관리 도구로는 기각. 빠르게 쓸 수 있지만 프로젝트·ADR·트러블슈팅을 연결한 자체 정보 구조와 사이트 코드 소유권을 약화한다. 발행한 글의 요약 유통 채널로는 사용할 수 있다.

## 위험과 완화

- 카테고리나 프로젝트 slug를 변경하면 관계가 자동으로 따라가지 않는다. `verify-content-links.mjs`가 유효하지 않은 참조를 CI에서 차단한다.
- GitHub App 자격증명이 필요하다. Vercel 환경 변수로만 보관하고 저장소·로그·공개 문서에는 값을 남기지 않는다.
- 관리자 번들이 크다. 공개 페이지와 분리되어 독자의 다운로드에는 포함되지 않는다.
- GitHub 장애 시 웹 편집을 사용할 수 없다. 로컬 Markdown 편집 경로와 초안 생성기를 계속 유지한다.

## 재검토 조건

- 여러 명의 실시간 편집과 세밀한 역할 권한이 필요함
- 모바일 편집 경험이 반복적으로 병목이 됨
- 미디어 자산 규모가 Git 저장소 운영에 부적합해짐
- GitHub API나 Keystatic 유지보수 정책이 개인 블로그 요구와 맞지 않음
