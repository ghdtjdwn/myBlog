# ADR-0005: GitHub와 블로그를 하나의 포트폴리오 진입점으로 운영한다

- 상태: 채택
- 결정일: 2026-07-15

## 맥락

채용 과정에서는 GitHub URL과 포트폴리오·블로그 URL을 서로 다르게 요구한다. 기존 GitHub
Pages 포트폴리오, 긴 프로필 README, 새 Astro 블로그가 동시에 존재하면 어느 정보가 최신인지
알기 어렵고 프로젝트 설명과 수치도 쉽게 어긋난다. GitHub 활동량만 강조하면 실제 문제 해결,
기여 경계와 검증 근거가 가려질 수 있다.

## 결정

GitHub 프로필과 `seongju.vercel.app`을 두 개의 공식 진입점으로 운영하되 역할을 분리한다.

- GitHub 프로필은 한 문단 소개, 3~4개 대표 작업, 기술, 교육·자격과 연락처를 빠르게 확인하는
  이력서형 색인이다.
- 블로그는 프로젝트별 문제, 개인 역할, 설계, 검증, 한계와 원본 기록 링크를 설명하는 사례집이다.
- 저장소 README, ADR, 작업 로그와 트러블슈팅은 재현 가능한 근거의 원본이다.
- 변동하는 commit·PR 합계와 활동량은 성과 지표로 사용하지 않는다.
- 비공개 학습 저장소와 기여 귀속이 불명확한 협업 저장소는 공개 프로필에서 링크하지 않는다.
- 프로젝트 이름이 바뀌어도 기존 연동을 보호해야 하는 slug와 조직 handle은 안정적으로 유지한다.

기존 GitHub Pages 포트폴리오는 연결을 제거한 뒤 Pages를 비활성화하고 저장소를 아카이브한다.
삭제 대신 아카이브를 선택해 이력을 보존하면서도 공식 포트폴리오로 오인되지 않게 한다.

## 대안

- 블로그만 사용: GitHub 제출을 요구하는 채용 과정에서 첫 화면의 정보 전달력이 부족하다.
- GitHub README에 모든 사례를 복제: 두 위치의 긴 본문과 수치가 빠르게 어긋나고 검토 시간이
  길어진다.
- 기존 Pages를 리다이렉트용으로 유지: 세 번째 운영 표면과 배포 설정을 계속 관리해야 하고
  오래된 검색 결과가 남는다.
- 과거 활동 이력을 다시 작성: 공개 기여 그래프를 인위적으로 정리하는 비용과 위험이 크고 실제
  엔지니어링 설명을 개선하지 않는다.

## 결과와 재검토

두 URL 중 어느 것을 제출해도 동일한 대표 작업과 연락처를 찾을 수 있고, 상세 설명은 블로그,
구현 근거는 GitHub로 자연스럽게 이어진다. 프로필 정보 변경 시 두 표면을 함께 검토한다. 개인
도메인, 이력서 또는 LinkedIn이 추가되면 새 진입점을 늘리기보다 이 구조 안의 링크만 갱신한다.
