# 프로젝트 작업 로그와 트러블슈팅 운영 방식

## 결정

프로젝트 기록은 GitHub 원본, 블로그 해설, 프로젝트 상세 색인의 세 층으로 운영한다.

```text
프로젝트 저장소
  ├─ Issues / Projects                 진행 중 작업과 다음 마일스톤
  ├─ docs/worklog/YYYY-MM-DD-topic.md  완료한 작업 단위와 실제 검증
  ├─ docs/adr/NNNN-topic.md            중요한 선택과 대안
  └─ docs/troubleshooting/slug.md      재현 가능한 실패·원인·해결
              ↓ 선별·재구성
기술 블로그 글                         문제, 판단, 구현, 실패, 검증, 한계
              ↓ 연결
프로젝트 상세 페이지                   현재 상태와 원문·블로그 링크 색인
```

모든 일지를 블로그에 복사하지 않는다. 저장소는 코드 변경과 같은 이력에서 사실을 확인하는 원본이고, 블로그는 채용 담당자와 개발자가 하나의 기술 문제를 이해할 수 있도록 맥락을 다시 구성하는 곳이다.

## 저장 위치별 역할

### GitHub Issues와 Projects

- 아직 끝나지 않은 작업, 버그, 실험 가설과 다음 마일스톤을 추적한다.
- 진행 중 상태와 담당 범위를 보여준다.
- 완료 후 사실 기록을 문서로 옮겼다면 Issue에서 해당 문서와 PR을 연결한다.

### 작업 로그

작은 기능을 포함한 완료 단위를 날짜순으로 남긴다. 최소 필드는 날짜, 목적, 중요한 변경, 영향 범위, 실제 수행한 검증, 전달 결과, 결정과 남은 작업이다. 실행하지 않은 테스트나 예상 성과를 쓰지 않는다.

### ADR

구조·인프라·데이터 모델처럼 되돌리는 비용이 있는 선택에만 작성한다. 문제, 선택한 안, 실제 검토한 대안, 판단 기준, 결과와 재검토 조건을 포함한다. 단순 라이브러리 추가마다 만들지 않는다.

### 트러블슈팅

실제 장애, 인증·동시성·정합성·외부 API·성능·배포 실패처럼 재사용 가능한 학습이 있을 때 작성한다. 기대와 실제, 영향, 재현, 가설, 증거, 원인, 대안, 해결, 검증, 재발 방지, 남은 위험을 포함한다. 루틴 작업을 장애처럼 꾸미지 않는다.

### 블로그

다음 중 하나에 해당할 때 원문을 그대로 복사하지 않고 독립 글로 다시 쓴다.

- 성능 병목을 실행계획·부하 테스트로 좁힌 사례
- 실제 배포·인증·세션·동시성·데이터 정합성 장애
- 큰 아키텍처 선택과 포기한 대안
- AI 평가, guardrail과 결정론 경계
- 공모전의 핵심 가설, 피벗과 제한 시간 안의 범위 결정
- 팀 프로젝트의 계약·리뷰·기여 경계를 설명할 수 있는 사례

글은 사용자 문제 → 제약과 개인 기여 → 선택지 → 구현 → 실패 → 검증 → 결과와 한계 → 다음 작업 순서로 구성하고, 공개 가능한 GitHub 문서·Issue·PR·commit을 연결한다.

## 공개 범위

- 공개 저장소: 원문 링크를 프로젝트 상세에 직접 연결한다.
- 비공개 팀 저장소: 팀원 코드, 내부 URL과 데이터를 공개하지 않고 개인 기여만 익명화해 요약한다.
- 작업 중 프로젝트: `진행 중`과 다음 검증을 명시하고 계획을 결과처럼 쓰지 않는다.
- 실패한 대회·실험: 사실과 배운 점이 확인되면 기록하되 수상·사용자 성과를 암시하지 않는다.

## 각 저장소에 새로 적용할 최소 구조

새 프로젝트는 다음 디렉터리를 기본으로 사용한다.

```text
docs/
  worklog/
  adr/
  troubleshooting/
```

기존 프로젝트는 이미 있는 `WORKLOG.md`, `TROUBLESHOOTING.md`, `docs/adr`를 유지해 불필요한 대규모 이동을 만들지 않는다. 새 기록부터 위 구조를 적용하고 프로젝트 README에서 세 위치를 링크한다.
